Deep Dive / Featured Project

Notification Pipeline

Python · Celery · Redis · SendGrid · PostgreSQL

The problem

The original product sent emails synchronously inside the HTTP request cycle. A slow SendGrid response meant the API endpoint would hang for 2–4 seconds — and if SendGrid was down, the user got a 500 error for actions that had nothing to do with email.

The task was to pull all notification delivery out of the request path and into a reliable async system, without losing events and without duplicating deliveries.

Architecture / request flow

An event fires in the API — an order is placed, a payment clears, an account is flagged. The API handler writes a notification record to PostgreSQL and enqueues a task to Redis. It then returns immediately. Celery workers pick up tasks from the queue, render the correct template, and call the relevant delivery channel. The result is written back to the notification record.

API Request → DB write → Redis enqueue → Celery worker → SendGrid / Push

Key endpoints

POST /notifications/trigger Internal — fires an event, enqueues delivery
GET /notifications/ Returns paginated list for the authed user
PATCH /notifications/:id/read Marks a notification as read
GET /notifications/preferences User's channel preferences
PATCH /notifications/preferences Update opt-in/out per event type

One challenge — duplicate deliveries

Celery's retry mechanism re-enqueues a task when a worker crashes mid-execution. With transactional email, a retry that succeeded on the first attempt but crashed after the send call would fire the same email twice. Users were getting duplicate order-confirmation emails during a stress test.

How I solved it

I added an idempotency_key column to the notification record — a hash of (user_id, event_type, event_id). Before making the SendGrid call, the worker attempts to flip a dispatched flag on the record using a conditional update (UPDATE … WHERE dispatched = false RETURNING id). If the update returns no rows, the notification was already dispatched and the worker exits early. The dispatch and the DB flag update happen in the same database transaction, making double-delivery structurally impossible even under concurrent retries.
