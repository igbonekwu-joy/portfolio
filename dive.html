Deep Dive / Featured Project

Retry Engine

Node.js · Express.js · Sqlite · Background Worker

The problem

In fintech and e-commerce, the backend constantly makes calls to external APIs, and it sometimes fails because of various reasons, like unreliable network, overloaded servers, and third party API downtime. If it is retried too fast, it makes things worse. If it is not retried at all, work is lost.

The retry engine solves this by accepting a request, returning immediately, and retrying the task in the background until it succeeds, with an exponential backoff and jitter strategy. It spaces out retries intelligently, giving the external services time to recover.

Architecture / request flow

The client makes a request --> Express API validates the payload, and saves the job to sqlite with a pending status --> API returns 202 Accepted to the client immediately --> A background worker continuously polls the database for pending jobs, picks one up, and attempts to execute it --> If the job succeeds, the worker updates the record to completed. If it fails with a retryable error, it calculates the next retry time using exponential backoff with jitter, updates the record with the new retry time, and moves on to the next pending job. If it fails with a non-retryable error or exceeds max retries, it marks the record as failed and moves on.

API Request → Validate Payload → DB Write (Pending Job) → 202 Accepted → Worker Polls DB → Execute Job → DB Update (Completed / Retry / Failed)

Key endpoints

POST /request Submit a new job for the worker to process
GET /requests/:id Returns details for a specific request and its full attempt history
GET /requests?status=:status Lists requests filtered by status

Technical Challenge: Duplicate Job Processing

The worker kept processing the same jobs multiple times, like when a job is already being processed but hasn't been marked as such in the database yet, the worker picks it up again and starts another attempt in parallel, leading to duplicate rows and potential issues in the future.

How I solved it

I locked the job synchronously before doing anything async. I updated the status to 'processing' immediately. Since better-sqlite3 is synchronous, the UPDATE status = 'processing' runs and commits instantly, before any await. By the time the next worker tick runs, the row is already processing and the SELECT query ignores it.
